# I18N (Localization)

Last updated: October 25, 2025 15:46

## Overview
- The app uses a simple context-driven i18n: `SettingsProvider` in `contexts/settings-context.tsx`.
- Exposes:
  - `language`, `setLanguage(lang)`
  - `currency`, `setCurrency(cur)`
  - `t(key: string): string` translation helper
  - `formatCurrency(value, opts?)`

## Where translations live
- Source of truth: `contexts/settings-context.tsx`
- The `translations` object maps a key to per-locale strings, e.g.:
  ```ts
  Home: { en: 'Home', ko: '홈', ja: 'ホーム', es: 'Inicio', zh: '首页' }
  ```
- If a key is missing for a locale, `t(key)` falls back to `en` (then the key name).

## How to add/modify a translation
1. Open `contexts/settings-context.tsx`.
2. Add a new entry under `translations` with all needed locales.
3. Replace any hardcoded UI string with `t('YourKey')` in the component.
4. If it’s a title in a Tab/Stack layout, ensure the container remounts on language change (see below).

Examples:
- Adding a new label:
  ```ts
  NewFeature: { en: 'New Feature', ko: '새 기능', ja: '新機能', es: 'Función nueva', zh: '新功能' }
  ```
- Using it in a component:
  ```tsx
  import { useSettings } from '@/contexts/settings-context';
  const { t } = useSettings();
  <ThemedText>{t('NewFeature')}</ThemedText>
  ```

## Currency formatting
- Use `formatCurrency(amount, { currency?: 'USD' | 'EUR' | ... })` for prices/amounts.
- Defaults to the selected `currency` and `language`.

## Layout titles and remounting
- Expo Router derives titles at render-time. To guarantee header/tab updates when language changes, add a `key` on layout components based on `language`:
  ```tsx
  const { t, language } = useSettings();
  <Stack key={`settings-stack-${language}`}> ... </Stack>
  <Tabs key={`tabs-${language}`}> ... </Tabs>
  ```
- Set titles with `t('...')` inside `<Stack.Screen options={{ title: t('...') }} />`.

## Tips
- Keep keys short and descriptive (e.g., `ReportProblem`, `HotCommunityPosts`).
- Reuse keys for identical phrases (`Risk`, `Popular`, `Falling`).
- Avoid embedding variables in strings when possible; compose with JSX or simple concatenation.

## Policy: No hard-coded UI strings
- Do not hard-code user-facing text in components. Always use `t('Key')`.
- Titles, placeholders, button labels, refresh titles, and section headers must be localized.
- If a string is truly temporary during development, add a TODO and replace it with `t(...)` before merging.

### Common pitfalls
- TextInput placeholders: never write `placeholder="..."`; use `placeholder={t('...')}`.
- RefreshControl title: avoid `title="..."`; use a localized key like `t('RefreshTitle')`.
- Composite strings: avoid concatenating literals; add keys or compose with variables.
- Page descriptions/details: never hard-code explanatory copy inside screens. Use `t('...Description')` keys.

### How to detect hard-coded strings
- Search for raw string props:
  - `title="..."`, `label="..."`, `placeholder="..."`
- Search for text nodes inside components like `<ThemedText>Some text</ThemedText>` that are not `t('...')`.
- Example to fix: RefreshControl `title="Stonket!"` → use `t('AppName')` or `t('RefreshTitle')`.
